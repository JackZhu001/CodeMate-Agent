10 分钟理解 CodeMate Agent

一个面向真实代码仓库的终端 AI 工程助手：能读、能查、能改、能追踪上下文， 并通过规划、记忆、压缩、心跳机制把复杂任务稳定推进。相比“纯聊天式助手”， CodeMate 更强调工程执行链路闭环：任务拆解 → 工具执行 → 状态追踪 → 结果校验。

项目概览

CodeMate 的目标不是一次性问答，而是完整的工程执行流。

定位清晰

聚焦“真实代码仓库”场景，强调可执行、可追踪、可恢复，而不是纯聊天。

新人友好

内置命令体系（/help /compact /heartbeat /init）与进度可视化，降低上手门槛。

工程化落地

支持工具调用、任务分解、分块写入、会话落盘，适合“边做边验”的真实开发流程。

面向发布

输出可直接用于文档、网页、项目介绍等对外物料，并保留一致性检查路径。

核心能力

从理解仓库到交付产物的关键能力闭环。

1) 项目理解与检索

  • 目录/文件/代码搜索工具联动
  • 支持 MCP/外部检索与本地文档互补
  • 可沉淀为 research.json 供后续引用

2) 任务规划与执行

  • 复杂任务自动生成 TODO 计划
  • 支持阶段更新、完成判定与干预
  • 可避免“只思考不交付”

3) 上下文工程

  • Micro Compact：工具输出分级裁剪
  • Auto Compact：超阈值自动压缩
  • Manual Compact：/compact 手动触发

4) 记忆系统

  • BM25 召回长期记忆片段
  • codemate.md 注入项目画像与偏好
  • 跨会话降低重复解释成本

5) 可观测与稳态控制

  • Heartbeat + Watchdog 超时告警
  • 循环检测与连续失败干预
  • Trace / Metrics 落盘可回放

6) 可靠文件产出

  • write_file_chunks / append_file_chunks 分块写入
  • 降低长内容参数截断导致失败
  • 支持回读自检确保产物可用

典型实战场景

不是抽象能力列表，而是新人第一周就会遇到的真实任务切片。

场景 A：接手旧仓库做改造

  • 先读 README/报告文档建立项目心智模型
  • 再用 rg/glob 定位关键入口与调用链
  • 输出改造计划并按 TODO 阶段推进

场景 B：大文档/网页生成

  • 先检索与对齐信息来源，沉淀 research.json
  • 使用 write_file_chunks/append_file_chunks 分块落盘
  • 完成后 read_file 回读，执行一致性自检

场景 C：线上问题复盘

  • 结合 trace/metrics 快速复盘失败链路
  • 利用心跳告警定位“卡住轮次”与超时步骤
  • 修复后用测试回归，避免同类问题重现

架构 / 流程

结构化组件 + 循环执行模型，保证“能做事且能收敛”。

架构分层

  1. 交互层CLI（prompt-toolkit + Rich）
  2. 控制层Agent + Planner + Loop Guard
  3. 模型层LLM Client（MiniMax 优先）
  4. 执行层ToolRegistry + File/Search/Shell/Task/Skill
  5. 状态层Context / Memory / Trace / Metrics

标准执行流程

  1. Step 1接收用户任务并判定复杂度
  2. Step 2复杂任务自动生成 TODO 计划
  3. Step 3调用 LLM 决策工具调用
  4. Step 4执行工具并回填结果
  5. Step 5必要时进行上下文压缩与记忆检索
  6. Step 6心跳监控 + 超时告警 + 循环干预
  7. Step 7产物落盘并回读自检
  8. Step 8输出最终交付说明与检查项

注意事项

给新人最关键的使用边界与避坑建议。

环境与模型

  • 建议使用 `python -m codemate_agent.cli` 启动
  • .env 至少配置：API_PROVIDER / BASE_URL / API_KEY / MODEL
  • 默认模型为 MiniMax-M2

MCP 测试策略

  • 外部检索建议设置预算（次数 + 超时）
  • MCP 失败要能降级到本地文档
  • 先产出 research.json，再进入页面生成

长内容写入

  • 不要一次性写入超长 HTML/CSS
  • 使用 chunk 工具分段写入
  • 每次写入后建议 read_file 回读校验

安全与可观测

  • run_shell 属于危险工具，需谨慎确认
  • 关注心跳告警与连续失败提示
  • 通过 trace/metrics 做复盘与优化

新人上手路线（建议照此执行）

把“会用”变成“能稳定交付”的四步法。

01

先理解项目边界

读 README、PROJECT_REPORT、PROJECT_ANALYSIS、WORKFLOW，明确当前架构与目标，不要直接开改。

02

再建立任务计划

复杂任务必须有 TODO 分解，阶段性更新进度，避免“做了很多但没有可验收结果”。

03

执行时保留可观测性

关注心跳告警、循环检测与失败计数；必要时缩小步长，优先保证每一步都可回放。

04

交付前做一致性检查

产物回读 + 关键结论对照文档来源；说明文件清单、作用和检查项，确保团队可复用。
